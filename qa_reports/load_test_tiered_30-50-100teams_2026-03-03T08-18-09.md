## 8. Analysis & Recommendations

### Architecture Under Test

```
Clients (30→50→100) ──► Nginx LB :80
                              │
                              ▼
                       api1 (Express + WS)
                              │
              ┌───────────────┴───────────────┐
              │                               │
       /submit  (direct spawn)         /run  (Bull → Redis)
              │                               │
        Executor proc                  Worker ×3 pods
           (per req)                   (shared pool)
```

### Observations

1. **Error ramp:** 68 more failures at 100 teams vs 30 teams. Indicates capacity pressure — consider adding api1 replicas.

2. **System Errors (5xx):** 11 across 180 submissions. May indicate OOM when spawning compiler processes or PgBouncer connection exhaustion.

3. **Timeouts:** 64 requests timed out (90s limit). The compile+judge path spawns OS processes; under heavy load the server can exhaust process or memory limits.

### Recommendations

| Priority | Recommendation |
|---|---|
| High | Scale api1 to **2+ replicas** for contests with >50 teams |
| High | Increase `ulimit -n` in api1 container (raise OS file descriptor limit) |
| Medium | Add a **submission queue** (Bull) for `/submit` to decouple compiler spawn from HTTP |
| Medium | Scale Bull workers from 3 → **5–8** pods for heavy /run traffic |
| Medium | Use **Redis result caching** — same code+problem = instant cached verdict |
| Low | Increase `PgBouncer pool_size` from 30 → 60 for >80 teams |
| Low | Pre-warm containers 5 min before contest start |
