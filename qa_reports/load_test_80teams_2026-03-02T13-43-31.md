## 8. Analysis & Recommendations

### Architecture Under Test

```
Client (80 concurrent) ──► Nginx LB :80
                              │
                              ▼
                         api1 (Express + WebSocket)
                              │
                    ┌─────────┴──────────┐
                    │                    │
             /submit path         /run path
        (direct compile)     (Bull Queue → Redis)
                    │                    │
             Executor proc       Worker ×3 pods
                 (per req)        (shared pool)
```

### Observations

1. **Timeouts:** 93 requests timed out. The compile+judge path spawns OS processes per request. Under 80 concurrent submits, the server can exhaust OS process limits or memory, causing job hangs.

2. **Network Errors:** 307 connection-level failures. The Nginx upstream may have rejected connections when the api1 container's event loop was saturated.

### Recommendations

| Priority | Recommendation |
|---|---|
| High | Scale api1 to **2+ replicas** behind Nginx when running contests with >40 teams |
| High | Raise `worker_processes` limit in Nginx and increase `ulimit -n` in the api1 container |
| Medium | Add a **submission queue** (Bull) for `/submit` to decouple compiler spawn from HTTP thread |
| Medium | Increase Bull worker replicas from 3 to **5–8** for heavy /run load |
| Medium | Use **Redis submission cache** aggressively — same code+problem = instant result (already implemented) |
| Low | Add **connection pooling** for DB reads (`PgBouncer pool_size` = 30→60) |
| Low | Pre-warm Docker containers before contest start (`docker compose up` 5 min before) |
